# Протокол работы ИИ-агентов с репозиторием

Файл обязателен для ИИ-агентов, изменяющих `AlexandrShapkin/knowledge`.

## Источник истины

Основная и публикуемая ветка — `main`.

Перед изменениями агент обязан:

1. прочитать актуальный `AGENTS.md`;
2. получить актуальную `main`;
3. прочитать изменяемые файлы из `main`;
4. создать отдельную ветку от актуальной `main`.

Нельзя начинать новую задачу от старой рабочей ветки или устаревшего commit SHA.

## Ветки и Pull Request

Для каждой логически связанной задачи создаётся отдельная короткоживущая ветка.

Рекомендуемые имена:

```text
docs/ssh-agent-note
fix/content-links
feat/wikilinks
refactor/quartz5-tooling
```

Процесс:

```text
main → рабочая ветка → изменения → проверки → PR → squash merge → проверка main
```

Прямая запись в `main` не используется без явного требования пользователя.

Название PR и итоговый squash-коммит оформляются как Conventional Commit.

## Границы Quartz и проекта

- `quartz/` и `docs/` содержат upstream Quartz 5;
- `quartz.config.yaml` и `quartz.lock.json` задают конфигурацию и плагины;
- `content/` содержит базу знаний;
- `scripts/` содержит проектную автоматизацию;
- верхнеуровневые Markdown-файлы описывают правила проекта.

Не следует исправлять проектные процессы патчами внутри `quartz/`, если задачу можно решить конфигурацией, плагином или скриптом вне upstream-кода.

## Контент

Перед изменением `content/` агент читает [`CONTENT_RULES.md`](./CONTENT_RULES.md).

Обязательные правила:

- Markdown-файл начинается с YAML frontmatter;
- `title` и непустой список `tags` обязательны;
- теги пишутся в lowercase;
- заметка раскрывает одну законченную тему;
- внутренние ссылки оформляются относительными Obsidian wikilinks;
- внешние URL остаются Markdown-ссылками;
- вложения находятся в локальном `!assets` и встраиваются через `![[...]]`;
- структурные изменения отражаются в ближайшем `index.md`.

Поддерживаемые примеры:

```markdown
[[Соседняя заметка]]
[[../Сети/DNS|Проверка DNS]]
[[Соседняя заметка#Раздел]]
[[Соседняя заметка#^block-id]]
![[!assets/Схема.png|Схема 800x600]]
![[Соседняя заметка#Раздел]]
```

Для `.md` и `.mdx` расширение обычно не указывается. Для `.canvas`, `.base` и вложений расширение сохраняется. Используется `/`, а не обратный слеш.

Quartz и валидатор настроены на `markdownLinkResolution: relative`. В Obsidian этому соответствует **New link format → Relative path to file**.

Существующие внутренние Markdown-ссылки поддерживаются для совместимости, но новые внутренние связи создаются как wikilinks. Для миграции используется:

```bash
npm run content:wikilinks
npm run content:wikilinks:check
```

## Публикуемый набор

`configuration.ignorePatterns` в `quartz.config.yaml` — единый источник истины для исключённых путей. Проектные валидаторы используют эти же шаблоны.

Сейчас не публикуются `private`, `templates`, `.obsidian` и `!Meta`. Каталог `!assets` публикуется и не должен добавляться в `ignorePatterns`.

`draft: true` полностью фильтрует Markdown-страницу. `unlisted: true` скрывает страницу из навигации и индексов, но не запрещает доступ по прямому URL. Для полного исключения каталога применяется `ignorePatterns`.

## Индексы

Нативный FolderPage Quartz создаёт списки каталогов. Проектные `index.md` дополнительно используются для явной иерархии, описаний и связей графа.

При изменении структуры:

```bash
npm run content:indexes
npm run content:validate
```

Генерируемые индексы используют wikilinks. Ручные `index.md` не перезаписываются; автоматически обновляются только файлы с `generated: true`.

## Синхронизация

Владелец:

```bash
npm run sync
```

Контрибьютор:

```bash
npm run sync -- --contributor
```

Проектная команда выполняет validate, commit, fetch, rebase и push без force. При конфликте rebase останавливается, push не выполняется, а перед rebase создаётся страховочная ветка.

Нельзя рекомендовать `npx quartz sync` для этого репозитория: текущая реализация Quartz использует другой remote/branch workflow и force push.

## Безопасность Git

Без явного подтверждения пользователя не следует выполнять разрушительные операции, удаляющие незакоммиченные данные или переписывающие удалённую историю.

Перед изменением файла через GitHub API нужно получить актуальный blob SHA. Последовательные изменения одного пути выполняются строго по очереди.

При неоднозначности предпочтительны рабочая ветка, PR и squash merge.

## Проверки

Для изменений контента:

```bash
npm run content:validate
```

Для изменений автоматизации:

```bash
npm run sync:test
```

Для конфигурации, стилей, TypeScript и Quartz:

```bash
npm run install-plugins
npx tsc --noEmit
npx quartz build
```

Перед merge агент обязан проверить:

- diff и список изменённых файлов;
- mergeability PR;
- успешную установку плагинов;
- `content:validate`;
- `sync:test`;
- TypeScript;
- production build.

Если CI падает, агент исправляет причину, а не отключает проверку.

## Документация

При изменении поведения команды, конфигурации или контентного контракта одновременно обновляются:

- `README.md`;
- `CONTENT_RULES.md`;
- `CONTRIBUTING.md`;
- `SMART_SYNC.md` при изменении Git-процесса;
- `AGENTS.md`.

Документация должна описывать фактическое поведение текущего кода, а не историческую реализацию.
