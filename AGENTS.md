# Протокол работы ИИ-агентов с репозиторием

Этот файл обязателен для ИИ-агентов, изменяющих `AlexandrShapkin/knowledge`.

## Источник истины

Основная и публикуемая ветка — `v4`.

Перед изменениями агент обязан:

1. прочитать актуальный `AGENTS.md`;
2. получить актуальную `v4`;
3. прочитать изменяемые файлы из `v4`;
4. создать отдельную ветку от актуальной `v4`.

Запрещено начинать новую задачу от старой рабочей ветки, устаревшего commit SHA или ранее сохранённого состояния репозитория.

## Ветки

Для каждой задачи создаётся отдельная короткоживущая ветка. Одна ветка должна содержать только одну логически связанную задачу.

Рекомендуемый формат имени:

```text
<type>/<short-kebab-case-description>
```

`type` должен соответствовать назначению изменения и по возможности совпадать с типом Conventional Commit:

- `feat` — новая функциональность;
- `fix` — исправление ошибки;
- `docs` — документация и заметки;
- `refactor` — изменение структуры без изменения поведения;
- `test` — тесты;
- `ci` — GitHub Actions и другая CI-автоматизация;
- `build` — сборка и зависимости;
- `chore` — техническое обслуживание;
- `perf` — оптимизация производительности;
- `revert` — отмена ранее внесённого изменения.

Примеры:

```text
docs/ssh-agent-note
fix/smart-sync-conflict-report
feat/contributor-sync-mode
ci/delete-merged-branches
```

Правила именования:

- использовать lowercase;
- слова разделять дефисами;
- не использовать пробелы, кириллицу и специальные символы;
- имя должно отражать задачу, а не имя исполнителя;
- не включать секреты, персональные данные и временные значения;
- не переиспользовать уже слитую или закрытую ветку для новой задачи.

### Жизненный цикл ветки

```text
актуальная v4
    ↓
новая рабочая ветка
    ↓
изменения и проверки
    ↓
pull request в v4
    ↓
squash merge или закрытие PR
    ↓
удаление рабочей ветки
```

После завершения работы агент обязан удалить созданную им рабочую ветку, чтобы не засорять репозиторий.

Перед удалением необходимо убедиться, что:

- pull request слит либо окончательно закрыт;
- нужные изменения присутствуют в `v4` или больше не требуются;
- ветка не является `v4` и не относится к другой активной задаче.

Удаление удалённой ветки:

```bash
git push origin --delete <branch>
```

Удаление локальной ветки после перехода на `v4`:

```bash
git switch v4
git pull --ff-only origin v4
git branch -d <branch>
git fetch --prune origin
```

Для несмерженной ветки запрещено использовать `git branch -D` без явного подтверждения пользователя.

Если используемый API или коннектор не поддерживает удаление ветки, агент обязан:

1. проверить, не была ли ветка удалена автоматически после merge;
2. сообщить точное имя оставшейся ветки;
3. дать пользователю команду или действие для её удаления;
4. не утверждать, что ветка удалена, пока это не проверено.

Ветки из fork удаляются владельцем fork. Агент оригинального репозитория не должен пытаться удалять чужую ветку без соответствующих прав.

## Conventional Commits

Для коммитов, названий pull request и итогового squash-коммита используется формат Conventional Commits:

```text
<type>[optional scope][optional !]: <description>
```

Примеры:

```text
docs(agents): define branch lifecycle
fix(sync): preserve remote markdown sections
feat(sync)!: change contributor remote configuration
ci: validate pull requests
```

Правила:

- `type` пишется в lowercase;
- `scope` используется только когда он добавляет полезный контекст;
- описание должно быть кратким и однозначным;
- один коммит не должен смешивать несвязанные изменения;
- `feat` используется для новой функциональности;
- `fix` используется для исправления ошибки;
- breaking change обозначается `!` перед `:` либо footer `BREAKING CHANGE:`;
- тело коммита используется для объяснения причины и важных деталей;
- footer используется для breaking changes, ссылок на issues и других trailers;
- запрещены бессодержательные сообщения вроде `update`, `changes`, `fix stuff` или `work`.

Так как репозиторий использует squash merge, название PR должно соответствовать Conventional Commits: оно становится основой итогового коммита в `v4`.

Для небольших промежуточных коммитов также следует использовать Conventional Commits. Если технический инструмент создал неподходящий commit message, итоговый squash-коммит обязан быть исправлен.

## Обязательный процесс

```text
актуальная v4
    ↓
рабочая ветка
    ↓
минимальные изменения
    ↓
проверки
    ↓
pull request в v4
    ↓
проверка mergeability и diff
    ↓
squash merge
    ↓
проверка результата в v4
    ↓
удаление рабочей ветки
```

Прямая запись в `v4` не используется без явного требования пользователя.

Если `v4` изменилась во время работы, рабочая ветка обновляется от актуальной `v4`, конфликты разрешаются, временные средства удаляются, после чего PR проверяется повторно.

## Заметки

Контент находится в `content/`.

Агент обязан:

- сохранять структуру и стиль файла;
- не переписывать весь файл без необходимости;
- не удалять несвязанные разделы;
- использовать UTF-8;
- проверять наличие существующей заметки по теме;
- добавлять ссылку из подходящего индекса, если это действительно улучшает навигацию;
- проверять внутренние ссылки и относительные пути;
- сохранять корректный frontmatter.

Теги пишутся в lowercase. Пустые элементы запрещены. Для SSH-заметок используется тег `ssh`.

### Внутренние ссылки

Во всех заметках агент обязан использовать только обычные относительные Markdown-ссылки.

Правильный формат:

```markdown
[Диагностика SSH](Диагностика%20SSH.md)
```

Ссылка на файл в другом каталоге:

```markdown
[SSH Config](SSH%20ключи/SSH%20Config.md)
```

Ссылка на раздел файла:

```markdown
[Проверка подключения](Диагностика%20SSH.md#проверка-подключения)
```

Ссылка на раздел текущей заметки:

```markdown
[Диагностика](#диагностика)
```

Правила:

- путь указывается относительно файла, в котором находится ссылка;
- пробелы в пути обязательно кодируются как `%20`;
- регистр символов в пути должен совпадать с фактическим именем файла;
- расширение `.md` для ссылок на заметки указывается явно;
- перед добавлением ссылки нужно проверить существование целевого файла;
- для изображений используется обычный Markdown-синтаксис `![Описание](путь/изображение.png)`;
- пробелы и специальные символы в путях изображений также должны быть URL-кодированы;
- агент не должен создавать абсолютные файловые пути или ссылки от корня файловой системы.

Запрещены Obsidian wiki-ссылки и embeds:

```markdown
[[Диагностика SSH]]
![[Схема.png]]
```

ИИ-агент не должен добавлять новые wiki-ссылки, даже если Obsidian умеет их обрабатывать. При редактировании существующего фрагмента агент должен привести затронутые им ссылки к Markdown-формату.

Массово преобразовывать все ссылки в несвязанных заметках запрещено, если пользователь отдельно не запросил миграцию ссылок.

## Пользовательская автоматизация

Каталог `quartz/` обновляется из `jackyzha0/quartz`. Пользовательские скрипты размещаются в `scripts/`, а команды — в `package.json`.

Запрещено исправлять синхронизацию патчами внутри `quartz/cli`, потому что `npx quartz update` может их затереть.

## Синхронизация

Владелец:

```bash
npm run sync
```

Контрибьютор:

```bash
npm run sync -- --contributor
```

Для оригинального репозитория базы знаний используется remote `knowledge-upstream`. Имя `upstream` зарезервировано Quartz.

Не следует рекомендовать `npx quartz sync` для получения заметок.

Автоматическое объединение не гарантирует смысловую корректность. После неоднозначного конфликта нужно проверить итоговый файл и отчёт в `.git/repository-sync/`.

## Деплой

`.github/workflows/deploy.yml` запускается после push в `v4`.

Основной путь автоматизированного изменения:

```text
ветка → PR → merge в v4 → deploy
```

Не следует рассчитывать, что push через стандартный токен одного GitHub Actions workflow запустит следующий workflow.

Временный workflow допустим только как техническая мера в рабочей ветке. До PR он должен быть удалён из итогового diff.

## Безопасность Git

Без явного подтверждения пользователя запрещено выполнять или рекомендовать как обычное решение:

```bash
git reset --hard
git clean -fd
git push --force
```

Перед изменением существующего файла через API нужно получить его актуальный blob SHA. Последовательные изменения одного пути нельзя выполнять параллельно.

## Проверки

Для документации и заметок:

- проверить frontmatter;
- проверить Markdown-ссылки и существование целевых файлов;
- проверить URL-кодирование пробелов в путях;
- проверить отсутствие случайных удалений;
- выполнить `npx quartz build`, если доступно.

Для синхронизации:

```bash
npm run sync:test
```

Для Quartz и TypeScript:

```bash
npm run check
```

Если проверку выполнить нельзя, агент обязан указать это.

## Итоговое сообщение

Нужно сообщить:

- что изменено;
- какие файлы затронуты;
- имя рабочей ветки;
- номер PR;
- был ли PR слит;
- итоговый commit SHA;
- была ли рабочая ветка удалена;
- какие проверки выполнены;
- команду `npm run sync` для получения изменений.
