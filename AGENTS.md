# Протокол работы ИИ-агентов с репозиторием

Этот файл обязателен для ИИ-агентов, изменяющих `AlexandrShapkin/knowledge`.

## Источник истины

Основная и публикуемая ветка — `v4`.

Перед изменениями агент обязан:

1. прочитать актуальный `AGENTS.md`;
2. получить актуальную `v4`;
3. прочитать изменяемые файлы из `v4`;
4. создать отдельную ветку от `v4`.

Рекомендуемое имя:

```text
ai/<краткое-название-задачи>
```

## Обязательный процесс

```text
актуальная v4
    ↓
рабочая ветка
    ↓
минимальные изменения
    ↓
проверки
    ↓
pull request в v4
    ↓
проверка mergeability и diff
    ↓
squash merge
    ↓
проверка результата в v4
```

Прямая запись в `v4` не используется без явного требования пользователя.

Если `v4` изменилась во время работы, рабочая ветка обновляется от актуальной `v4`, конфликты разрешаются, временные средства удаляются, после чего PR проверяется повторно.

## Заметки

Контент находится в `content/`.

Агент обязан:

- сохранять структуру и стиль файла;
- не переписывать весь файл без необходимости;
- не удалять несвязанные разделы;
- использовать UTF-8;
- проверять наличие существующей заметки по теме;
- добавлять ссылку из подходящего индекса;
- проверять Obsidian-ссылки и относительные пути;
- сохранять корректный frontmatter.

Теги пишутся в lowercase. Пустые элементы запрещены. Для SSH-заметок используется тег `ssh`.

## Пользовательская автоматизация

Каталог `quartz/` обновляется из `jackyzha0/quartz`. Пользовательские скрипты размещаются в `scripts/`, а команды — в `package.json`.

Запрещено исправлять синхронизацию патчами внутри `quartz/cli`, потому что `npx quartz update` может их затереть.

## Синхронизация

Владелец:

```bash
npm run sync
```

Контрибьютор:

```bash
npm run sync -- --contributor
```

Для оригинального репозитория базы знаний используется remote `knowledge-upstream`. Имя `upstream` зарезервировано Quartz.

Не следует рекомендовать `npx quartz sync` для получения заметок.

Автоматическое объединение не гарантирует смысловую корректность. После неоднозначного конфликта нужно проверить итоговый файл и отчёт в `.git/repository-sync/`.

## Деплой

`.github/workflows/deploy.yml` запускается после push в `v4`.

Основной путь автоматизированного изменения:

```text
ветка → PR → merge в v4 → deploy
```

Не следует рассчитывать, что push через стандартный токен одного GitHub Actions workflow запустит следующий workflow.

Временный workflow допустим только как техническая мера в рабочей ветке. До PR он должен быть удалён из итогового diff.

## Безопасность Git

Без явного подтверждения пользователя запрещено выполнять или рекомендовать как обычное решение:

```bash
git reset --hard
git clean -fd
git push --force
```

Перед изменением существующего файла через API нужно получить его актуальный blob SHA. Последовательные изменения одного пути нельзя выполнять параллельно.

## Проверки

Для документации и заметок:

- проверить frontmatter;
- проверить ссылки;
- проверить отсутствие случайных удалений;
- выполнить `npx quartz build`, если доступно.

Для синхронизации:

```bash
npm run sync:test
```

Для Quartz и TypeScript:

```bash
npm run check
```

Если проверку выполнить нельзя, агент обязан указать это.

## Итоговое сообщение

Нужно сообщить:

- что изменено;
- какие файлы затронуты;
- номер PR;
- был ли PR слит;
- итоговый commit SHA;
- какие проверки выполнены;
- команду `npm run sync` для получения изменений.
