# Протокол работы ИИ-агентов с репозиторием

Этот файл обязателен для любых ИИ-агентов, которые читают или изменяют репозиторий `AlexandrShapkin/knowledge`.

Цель протокола — гарантировать, что изменения:

- вносятся в актуальное состояние репозитория;
- не теряют локальные или удалённые правки;
- корректно попадают в ветку `v4`;
- запускают сборку и публикацию Quartz;
- нормально получаются на локальной машине через `npm run sync`;
- не ломаются после обновления исходного Quartz.

---

## 1. Источник истины

Основная и публикуемая ветка:

```text
v4
```

Перед любыми изменениями агент обязан:

1. Получить актуальное состояние `v4`.
2. Читать изменяемые файлы именно из актуальной `v4`.
3. Создать рабочую ветку от текущей `v4`.

Рекомендуемый формат имени ветки:

```text
ai/<краткое-название-задачи>
```

Допустимые примеры:

```text
ai/ssh-agent-note
ai/update-linux-tags
ai/fix-broken-links
```

Не следует начинать работу от старого commit SHA, ранее созданной ветки или локально запомненного состояния репозитория.

---

## 2. Стандартный порядок внесения изменений

ИИ-агент должен использовать следующий процесс:

```text
актуальная v4
    ↓
новая рабочая ветка
    ↓
изменения
    ↓
проверка diff и тестов
    ↓
pull request в v4
    ↓
проверка mergeability
    ↓
merge
    ↓
автоматический deploy Quartz
```

Прямая запись в `v4` не используется, если пользователь явно не потребовал обратного.

После создания pull request агент обязан проверить:

- что base-ветка — `v4`;
- что head-ветка содержит только изменения текущей задачи;
- что PR допускает merge;
- что в diff отсутствуют временные workflow, тестовые файлы и случайные изменения;
- что после merge изменения действительно присутствуют в `v4`.

Предпочтительный способ объединения небольших изменений:

```text
squash merge
```

---

## 3. Если ветка `v4` изменилась во время работы

Перед merge необходимо повторно проверить состояние PR.

Если PR больше не mergeable или рабочая ветка отстаёт от `v4`, агент должен:

1. Получить актуальную `v4`.
2. Объединить её с рабочей веткой обычным merge или rebase.
3. Разрешить конфликты без удаления изменений пользователя.
4. Повторно проверить diff.
5. Удалить все временные средства, созданные только для обновления ветки.
6. После этого выполнить merge PR.

Запрещено решать такую ситуацию через force push в `v4`.

---

## 4. Правила изменения заметок

Основной пользовательский контент находится в:

```text
content/
```

При изменении заметок агент обязан:

- сохранять существующую структуру и стиль файла;
- не переписывать весь файл без необходимости;
- не удалять пользовательские разделы, ссылки, изображения и frontmatter;
- использовать UTF-8;
- сохранять русскоязычный стиль заметок;
- проверять существующие связанные заметки перед созданием новой;
- не создавать дублирующую заметку, если тема уже раскрыта;
- добавлять ссылки на новую заметку из подходящего индексного файла;
- использовать существующий стиль Obsidian-ссылок;
- проверять корректность относительных путей и wiki-ссылок.

Перед обновлением существующего файла агент должен сначала прочитать его актуальную версию из рабочей ветки либо `v4`.

---

## 5. Frontmatter и теги

Рекомендуемый frontmatter заметки:

```yaml
---
title: Название заметки
draft: false
tags:
  - основной-тег
  - дополнительный-тег
---
```

Правила тегов:

- использовать lowercase;
- использовать уже существующие теги, когда это возможно;
- не создавать синонимы одного тега без необходимости;
- не оставлять пустое значение вида `-`;
- не добавлять слишком общие теги, не связанные с содержанием;
- для всех SSH-заметок использовать тег `ssh`;
- предметные теги добавлять по фактическому содержанию заметки.

Перед массовой расстановкой тегов агент должен посмотреть несколько соседних заметок и сохранить принятую в репозитории схему.

---

## 6. Quartz и пользовательские расширения

Каталог:

```text
quartz/
```

содержит исходный код upstream Quartz и может изменяться при:

```bash
npx quartz update
```

Поэтому собственную автоматизацию репозитория запрещено размещать внутри `quartz/`, если задача не относится непосредственно к изменению самого Quartz.

Пользовательские скрипты размещаются в:

```text
scripts/
```

Команды для них добавляются в:

```text
package.json
```

Документация пользовательской автоматизации размещается в корне репозитория или в отдельном пользовательском разделе документации.

Не следует исправлять поведение `npx quartz sync` патчами в `quartz/cli`, потому что такие изменения могут быть затёрты upstream-обновлением.

---

## 7. Синхронизация локальной и удалённой копии

Для пользовательской синхронизации применяется:

```bash
npm run sync
```

Описание реализации находится в:

```text
SMART_SYNC.md
```

ИИ-агент не должен рекомендовать `npx quartz sync` для получения изменений заметок из удалённого репозитория.

После merge изменений в `v4` пользователю следует сообщить команду:

```bash
npm run sync
```

Если локально уже выполняется merge, rebase, cherry-pick или revert, сначала необходимо завершить либо отменить текущую операцию.

Запрещено автоматически выполнять:

```bash
git reset --hard
git clean -fd
git push --force
```

без явного подтверждения пользователя и объяснения, какие данные могут быть потеряны.

---

## 8. Политика разрешения конфликтов

При работе через `npm run sync` Markdown-конфликты разрешаются трёхсторонним алгоритмом из:

```text
scripts/smart-sync.mjs
```

Базовые правила:

- независимые изменения объединяются;
- списки и теги объединяются без дублей;
- при неоднозначности сохраняются обе версии текста;
- перед синхронизацией создаётся резервная ветка;
- информация о неоднозначных решениях сохраняется в `.git/smart-sync/`.

ИИ-агент не должен считать автоматический merge семантически безошибочным. После сложного конфликта необходимо рекомендовать просмотр итоговой заметки.

---

## 9. Деплой Quartz

Публикация выполняется workflow:

```text
.github/workflows/deploy.yml
```

Он запускается после push в:

```text
v4
```

Обычный merge pull request в `v4` должен запускать сборку и публикацию Quartz.

Не следует рассчитывать, что push, выполненный GitHub Actions через стандартный `GITHUB_TOKEN`, автоматически запустит другой workflow. GitHub блокирует такие цепочки для предотвращения рекурсивных запусков.

Поэтому запрещён следующий основной сценарий:

```text
временный workflow
    ↓
workflow коммитит в v4
    ↓
ожидание автоматического запуска deploy workflow
```

Для изменений контента применяется PR → merge в `v4`.

Временный workflow допустим только как крайняя техническая мера. Он должен быть удалён из итогового diff до merge.

---

## 10. Проверки перед merge

Минимальная проверка для изменения заметок:

1. Frontmatter корректно закрыт `---`.
2. В тегах нет пустых элементов.
3. Markdown-кодовые блоки закрыты.
4. Новые внутренние ссылки ведут на существующие или одновременно создаваемые заметки.
5. В diff отсутствует случайное удаление больших частей файла.
6. Нет временных файлов и workflow.

Для изменения Smart Sync:

```bash
npm run sync:test
```

Для изменения TypeScript, компонентов или конфигурации Quartz:

```bash
npm run check
```

При возможности также выполнить:

```bash
npx quartz build
```

Если агент не мог выполнить проверку, он обязан прямо указать это в итоговом сообщении.

---

## 11. Правила работы через GitHub API и коннекторы

При использовании GitHub-инструментов агент должен:

1. Читать файл с явным `ref: v4` либо ref рабочей ветки.
2. Создавать рабочую ветку через `base_ref: v4`.
3. Выполнять последовательные изменения одного файла, используя актуальный blob SHA.
4. Не выполнять параллельные update/delete одного пути.
5. Перед merge проверять актуальный статус PR.
6. После merge проверять наличие результата в `v4`.

Если API не позволяет удобно изменить небольшой участок и требует полной замены файла, агент должен особенно внимательно сохранить всё исходное содержимое.

Запрещено использовать саморедактирующийся workflow только для обхода ограничений API, если изменение можно выполнить через ветку, PR или обычный Git.

---

## 12. Итоговое сообщение пользователю

После внесения изменений агент должен сообщить:

- что именно изменено;
- в какие файлы внесены изменения;
- номер pull request;
- был ли PR слит;
- commit SHA итогового merge;
- какие проверки выполнены;
- какую команду выполнить локально.

Стандартная команда получения изменений:

```bash
npm run sync
```

Не следует утверждать, что изменения опубликованы, если merge или deploy не были проверены.

---

## 13. Краткий обязательный алгоритм

Перед каждым изменением:

```text
прочитать AGENTS.md
получить актуальную v4
создать отдельную ветку
прочитать актуальные версии файлов
внести минимальные изменения
проверить diff
создать PR в v4
обновить ветку при необходимости
проверить mergeability
удалить временные средства
выполнить merge
проверить результат в v4
сообщить пользователю npm run sync
```
