## Назначение

SSH Agent позволяет хранить разблокированные приватные ключи в памяти и автоматически использовать их при SSH-аутентификации.

Основное преимущество заключается в том, что парольную фразу (passphrase) от приватного ключа требуется вводить только один раз после загрузки ключа в агент.

Без SSH Agent:

```text
ssh → ввод passphrase
git pull → ввод passphrase
git push → ввод passphrase
scp → ввод passphrase
```

С SSH Agent:

```text
ssh-add → ввод passphrase один раз

ssh
git pull
git push
scp
```

Все последующие операции используют уже загруженный ключ.

---

## Как работает SSH Agent

Упрощённая схема:

```text
┌─────────────┐
│ SSH Client  │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ SSH Agent   │
│ ключ в RAM  │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ SSH Server  │
└─────────────┘
```

Приватный ключ не покидает локальную систему.

SSH Agent выполняет криптографические операции от имени клиента и предоставляет результат SSH-клиенту.

---

## Проверка работы агента

Проверить наличие агента:

```bash
echo $SSH_AUTH_SOCK
```

Если переменная заполнена:

```text
/tmp/ssh-XXXXXX/agent.12345
```

то клиент знает, как связаться с агентом.

---

## Просмотр загруженных ключей

Список ключей:

```bash
ssh-add -l
```

Пример:

```text
256 SHA256:xxxxxxxxxxxxxxxxxxxxx work-laptop (ED25519)
```

Если ключей нет:

```text
The agent has no identities.
```

---

## Запуск агента

### Linux / macOS

Запуск вручную:

```bash
eval "$(ssh-agent -s)"
```

Пример результата:

```text
Agent pid 12345
```

После запуска появятся переменные окружения, необходимые для работы клиента с агентом.

---

### Windows

Проверить состояние службы:

```powershell
Get-Service ssh-agent
```

Запустить службу:

```powershell
Start-Service ssh-agent
```

Для автоматического запуска:

```powershell
Set-Service ssh-agent -StartupType Automatic
```

---

## Добавление ключа в агент

Стандартный ключ:

```bash
ssh-add ~/.ssh/id_ed25519
```

Ключ с произвольным именем:

```bash
ssh-add ~/.ssh/github
```

После ввода парольной фразы ключ будет храниться в памяти агента.

Проверить результат:

```bash
ssh-add -l
```

---

## Добавление нескольких ключей

Пример:

```bash
ssh-add ~/.ssh/github
ssh-add ~/.ssh/gitlab
ssh-add ~/.ssh/production
```

Просмотр:

```bash
ssh-add -l
```

Результат:

```text
256 SHA256:... github
256 SHA256:... gitlab
256 SHA256:... production
```

SSH будет выбирать подходящий ключ автоматически.

---

## Удаление ключей из агента

Удалить конкретный ключ:

```bash
ssh-add -d ~/.ssh/github
```

Удалить все ключи:

```bash
ssh-add -D
```

Проверка:

```bash
ssh-add -l
```

---

## Проверка использования агента

Подробный вывод SSH:

```bash
ssh -vvv user@host
```

Искать строки вида:

```text
get_agent_identities
```

или:

```text
Offering public key
```

Они показывают, какие ключи агент предоставляет клиенту.

---

## Ограничение времени хранения ключа

Ключ может автоматически удаляться из памяти через заданный интервал.

Пример:

```bash
ssh-add -t 1h ~/.ssh/id_ed25519
```

или:

```bash
ssh-add -t 3600 ~/.ssh/id_ed25519
```

Проверка:

```bash
ssh-add -l
```

После истечения времени ключ будет удалён автоматически.

## Использование с несколькими ключами

При наличии большого количества ключей иногда возникает ошибка:

```text
Too many authentication failures
```

Причина — агент пытается отправить серверу слишком много ключей.

Решение:

```sshconfig
Host prod
    IdentityFile ~/.ssh/prod
    IdentitiesOnly yes
```

Либо подключаться явно:

```bash
ssh -i ~/.ssh/prod user@host
```

Подробнее: [[SSH Config]]

## Автоматическая загрузка ключей

### Linux

Добавить в профиль оболочки:

```bash
eval "$(ssh-agent -s)"
```

Однако в современных дистрибутивах агент обычно запускается автоматически менеджером сессии.

Перед добавлением подобной настройки стоит проверить текущую конфигурацию системы.

---

### macOS

Добавить ключ:

```bash
ssh-add --apple-use-keychain ~/.ssh/id_ed25519
```

Система может сохранять парольную фразу в связке ключей Keychain.

Проверить настройки:

```text
~/.ssh/config
```

Пример:

```sshconfig
Host *
    AddKeysToAgent yes
    UseKeychain yes
```

---

### Windows

После включения службы:

```powershell
ssh-add $env:USERPROFILE\.ssh\id_ed25519
```

Ключ будет доступен всем программам, использующим встроенный OpenSSH.

## Использование с Git

Проверить наличие ключа:

```bash
ssh-add -l
```

Проверить подключение:

```bash
ssh -T git@github.com
```

После загрузки ключа Git сможет выполнять операции через SSH без постоянного запроса парольной фразы.

Подробнее: [[SSH для Git]]

## Agent Forwarding

SSH Agent можно пересылать на промежуточные серверы.

Схема:

```text
Laptop
   │
   ▼
Bastion
   │
   ▼
Production
```

При этом приватный ключ остаётся на локальном компьютере.

Подключение:

```bash
ssh -A user@bastion
```

Проверка на промежуточном сервере:

```bash
ssh-add -l
```

---

## Риски Agent Forwarding

Agent Forwarding удобен, но увеличивает поверхность атаки.

Если промежуточный сервер скомпрометирован, злоумышленник может использовать агент для выполнения аутентификации от имени пользователя до завершения сессии.

Поэтому:

- не использовать `ForwardAgent yes` глобально;
    
- включать только для доверенных серверов;
    
- отключать после необходимости.
    

Пример:

```sshconfig
Host bastion
    ForwardAgent yes
```

Не рекомендуется:

```sshconfig
Host *
    ForwardAgent yes
```

## Частые ошибки

### The agent has no identities

Причина:

```text
Ключи не загружены в агент.
```

Исправление:

```bash
ssh-add ~/.ssh/id_ed25519
```

---

### Could not open a connection to your authentication agent

Причина:

```text
Агент не запущен.
```

Исправление:

```bash
eval "$(ssh-agent -s)"
```

или запуск службы `ssh-agent` в Windows.

---

### Agent admitted failure to sign

Возможные причины:

- проблемы агента;
    
- повреждённый ключ;
    
- ошибки пересылки агента;
    
- аппаратные токены.
    

Попробовать:

```bash
ssh-add -D
ssh-add ~/.ssh/id_ed25519
```

---

### Too many authentication failures

Проверить:

```bash
ssh-add -l
```

Удалить лишние ключи:

```bash
ssh-add -D
```

Загрузить только нужный:

```bash
ssh-add ~/.ssh/production
```

---

## Практические рекомендации

- Использовать парольные фразы для приватных ключей.
    
- Загружать ключи через SSH Agent вместо хранения незашифрованных ключей.
    
- Использовать отдельные ключи для разных задач.
    
- Не включать глобальный `ForwardAgent`.
    
- Регулярно проверять содержимое агента через `ssh-add -l`.
    
- Ограничивать срок жизни чувствительных ключей через `ssh-add -t`.
