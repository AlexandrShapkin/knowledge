## Назначение

Для аутентификации по SSH-ключу сервер должен знать публичный ключ клиента.

Публичный ключ добавляется в файл:

```text
~/.ssh/authorized_keys
```

на удалённом сервере.

После этого пользователь сможет проходить аутентификацию с помощью соответствующего приватного ключа.

---

## Предварительные требования

На клиенте должна существовать пара SSH-ключей.

Проверить наличие ключей:

```bash
ls -l ~/.ssh
```

Обычно присутствуют файлы:

```text
id_ed25519
id_ed25519.pub
```

Если ключи отсутствуют, необходимо сначала создать их.

Подробнее: [Генерация ED25519 ключа](Генерация%20ED25519%20ключа.md)

---

## Способ 1. Использование ssh-copy-id

Предпочтительный способ.

Команда автоматически:
- создаёт каталог `.ssh`, если его нет;
- добавляет ключ в `authorized_keys`;
- выставляет необходимые права доступа;
- предотвращает большинство ошибок ручной настройки.

Команда:

```bash
ssh-copy-id user@host
```

Если используется нестандартный ключ:

```bash
ssh-copy-id -i ~/.ssh/github.pub user@host
```

Если SSH работает на нестандартном порту:

```bash
ssh-copy-id -p 2222 user@host
```

При первом подключении потребуется пароль пользователя на сервере.

После завершения можно проверить вход:

```bash
ssh user@host
```

---

## Способ 2. Ручное добавление ключа

Используется если `ssh-copy-id` отсутствует или недоступен.

Просмотреть содержимое публичного ключа:

```bash
cat ~/.ssh/id_ed25519.pub
```

Пример:

```text
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... work-laptop
```

Скопировать строку полностью.

На сервере открыть файл:

```bash
~/.ssh/authorized_keys
```

Добавить ключ отдельной строкой:

```text
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... work-laptop
```

Сохранить изменения.

---

## Способ 3. Добавление ключа одной командой

Удобно при наличии доступа по паролю.

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@host \
"mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```

После выполнения рекомендуется проверить права доступа.

---

## Создание каталога .ssh вручную

Если каталог отсутствует:

```bash
mkdir -p ~/.ssh
```

Назначить права:

```bash
chmod 700 ~/.ssh
```

---

## Настройка authorized_keys

Если файл отсутствует:

```bash
touch ~/.ssh/authorized_keys
```

Установить права:

```bash
chmod 600 ~/.ssh/authorized_keys
```

Проверить:

```bash
ls -ld ~/.ssh
ls -l ~/.ssh/authorized_keys
```

Ожидаемый результат:

```text
drwx------ .ssh
-rw------- authorized_keys
```

---

## Проверка добавленного ключа

Показать содержимое файла:

```bash
cat ~/.ssh/authorized_keys
```

Убедиться, что ключ присутствует.

Также можно проверить количество ключей:

```bash
wc -l ~/.ssh/authorized_keys
```

---

## Проверка входа по ключу

Подключиться:

```bash
ssh user@host
```

Если ключ используется корректно, пароль пользователя запрашиваться не будет.

Для подробной диагностики:

```bash
ssh -v user@host
```

Полный вывод:

```bash
ssh -vvv user@host
```

---

## Использование нестандартного ключа

Если ключ называется не `id_ed25519`:

```bash
ssh -i ~/.ssh/github user@host
```

Либо указать ключ в конфигурации SSH.

Пример:

```sshconfig
Host myserver
    HostName host.example.com
    User user
    IdentityFile ~/.ssh/github
```

Подробнее: [SSH Config](../SSH%20Config.md)

---

## Добавление нескольких ключей

Файл `authorized_keys` может содержать множество ключей.

Пример:

```text
ssh-ed25519 AAAA... admin-laptop
ssh-ed25519 AAAA... work-laptop
ssh-ed25519 AAAA... backup-laptop
```

Каждый ключ предоставляет доступ независимо от остальных.

---

## Удаление ключа

Открыть файл:

```bash
~/.ssh/authorized_keys
```

Удалить соответствующую строку.

После сохранения доступ по данному ключу будет немедленно прекращён.

---

## Ограничение возможностей ключа

Перед ключом можно указать ограничения.

Разрешить подключение только с конкретного адреса:

```text
from="192.168.1.10" ssh-ed25519 AAAA...
```

Запретить проброс портов:

```text
no-port-forwarding ssh-ed25519 AAAA...
```

Запретить выполнение команд:

```text
no-pty ssh-ed25519 AAAA...
```

Такие ограничения часто используются для сервисных и автоматизированных учётных записей.

---

## Добавление ключа для root

Подключиться под root:

```bash
sudo -i
```

Добавить ключ в:

```text
/root/.ssh/authorized_keys
```

Следует учитывать, что на многих серверах вход под root по SSH намеренно запрещён.

Проверить настройку:

```bash
grep PermitRootLogin /etc/ssh/sshd_config
```

---

## Типовые ошибки

### Permission denied (publickey)

Сервер не принял предоставленный ключ.

Возможные причины:
- ключ отсутствует в `authorized_keys`;
- используется другой приватный ключ;
- неверный пользователь;
- неправильные права доступа;
- SSH-клиент не отправляет нужный ключ.

Диагностика:

```bash
ssh -vvv user@host
```

---

### Bad owner or permissions

Некорректные права на файлы SSH.

Исправление:

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
```

---

### Authentication refused

Сервер отклонил авторизацию.

Проверить:

```bash
cat ~/.ssh/authorized_keys
```

и убедиться, что ключ добавлен полностью.

---

### Публичный ключ добавлен не тому пользователю

Частая ошибка.

Например:

```bash
ssh admin@host
```

но ключ находится в:

```text
/home/user/.ssh/authorized_keys
```

Для каждого пользователя используется собственный файл `authorized_keys`.

---

## Проверка со стороны сервера

Убедиться, что SSH-служба работает:

```bash
systemctl status ssh
```

или:

```bash
systemctl status sshd
```

Проверить журналы:

```bash
journalctl -u ssh
```

или:

```bash
journalctl -u sshd
```

## Рекомендуемый порядок действий

1. [Генерация ED25519 ключа](Генерация%20ED25519%20ключа.md).
2. Проверить наличие публичного ключа.
3. Добавить ключ через `ssh-copy-id`.
4. Проверить вход по SSH.
5. Настроить SSH Agent.
6. Добавить запись в `~/.ssh/config`.
7. При необходимости отключить парольную аутентификацию.
