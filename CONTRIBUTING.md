# Внесение изменений

Перед работой прочитайте [`CONTENT_RULES.md`](./CONTENT_RULES.md). Правила обязательны для всех файлов внутри `content/`.

## Подготовка fork

```bash
git clone https://github.com/USER/knowledge.git
cd knowledge
git remote add knowledge-upstream https://github.com/AlexandrShapkin/knowledge.git
npm ci
git fetch knowledge-upstream v4
git switch -c docs/my-change knowledge-upstream/v4
```

`origin` должен указывать на fork. Имя `upstream` зарезервировано Quartz, поэтому оригинальный репозиторий называется `knowledge-upstream`.

Одна ветка и один Pull Request должны соответствовать одной логически связанной задаче.

## Создание заметки

Перед созданием файла:

1. проверьте, нет ли существующей заметки по теме;
2. выберите ближайший тематический каталог;
3. определите полезные связи с другими заметками;
4. создайте Markdown-файл с понятным именем;
5. добавьте YAML frontmatter с `title` и `tags`.

```yaml
---
title: Название заметки
tags:
  - основной-тег
---
```

Заметка должна раскрывать одну тему и быть понятной без обязательного чтения соседних файлов.

## Каталоги и index.md

`index.md` требуется только непустому каталогу-разделу.

Каталог считается непустым, если в нём есть Markdown-файл или непустой дочерний раздел. Полностью пустой каталог является заготовкой структуры, не считается разделом, не требует индекса и не включается в родительскую навигацию.

После добавления первой заметки в пустой каталог создайте или обновите его `index.md` и добавьте ссылку из ближайшего родительского индекса.

Каталоги, название которых начинается с `!`, являются служебными и также не требуют `index.md`.

## Ссылки

Используются только относительные Markdown-ссылки:

```markdown
- [Диагностика SSH](Диагностика%20SSH.md)
- [Проверка DNS](../Сети/Проверка%20DNS.md)
```

Wiki-ссылки Obsidian запрещены:

```markdown
[[Диагностика SSH]]
![[Схема сети.png]]
```

Ссылка на заметку должна содержать `.md`, использовать URL-кодирование пробелов и вести на существующий файл.

## Вложения

Изображения и другие вложения хранятся в `!assets` той же директории, где находится использующая их заметка.

```text
content/Kubernetes/Ingress/
├── index.md
├── TLS через cert-manager.md
└── !assets/
    └── Схема выпуска TLS-сертификата.png
```

Имя вложения должно описывать содержимое. Имена вроде `Pasted image`, `image.png`, `screenshot.png`, `file.pdf` и `Untitled` нужно заменить до commit.

## Проверка

```bash
npm run content:validate
npx quartz build
```

Проверяются:

- `index.md` в непустых каталогах-разделах;
- пропуск полностью пустых и служебных каталогов;
- frontmatter, `title` и `tags`;
- внутренние Markdown-ссылки;
- связанность заметок;
- расположение и имена вложений.

Для изменений синхронизации выполните:

```bash
npm run sync:test
```

## Синхронизация ветки

```bash
npm run sync -- --contributor
```

Аварийный обход локальной проверки:

```bash
npm run sync -- --contributor --no-check
```

`--no-check` не отключает проверки Pull Request и deploy.

## Pull Request

Откройте Pull Request в `AlexandrShapkin/knowledge:v4`. Название должно соответствовать Conventional Commits:

```text
docs(ssh): add host key verification note
fix(content): repair broken Kubernetes links
```

В описании перечислите изменения и выполненные проверки.

## Контрольный список

- [ ] Заметка раскрывает одну тему и является законченной.
- [ ] Есть `title` и непустой список `tags`.
- [ ] Заметка связана с графом.
- [ ] В каждом непустом каталоге-разделе есть `index.md`.
- [ ] Полностью пустые каталоги оставлены без индекса.
- [ ] Ссылки ведут на существующие файлы.
- [ ] Вложения находятся в локальном `!assets`.
- [ ] Выполнены `npm run content:validate` и `npx quartz build`.

## Запрещённые действия

Без явной необходимости не используйте:

```bash
npx quartz sync
git push --force
git reset --hard
git clean -fd
```

Пользовательскую автоматизацию не размещают внутри `quartz/`, потому что `npx quartz update` может затереть изменения.
