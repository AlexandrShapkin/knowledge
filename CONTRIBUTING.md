# Внесение изменений

Этот документ описывает полный цикл работы с fork: настройку репозитория, создание заметки, проверку, синхронизацию и открытие Pull Request.

Перед началом работы прочитайте [`CONTENT_RULES.md`](./CONTENT_RULES.md). Этот документ является обязательной спецификацией для всех файлов внутри `content/`.

## Требования

Нужны Git, Node.js 22 или новее и npm 10.9.2 или новее.

```bash
git --version
node --version
npm --version
```

## 1. Создание и клонирование fork

Создайте fork `AlexandrShapkin/knowledge` в интерфейсе GitHub, затем клонируйте его:

```bash
git clone https://github.com/USER/knowledge.git
cd knowledge
```

`origin` должен указывать на ваш fork.

Добавьте оригинальный репозиторий под именем `knowledge-upstream`:

```bash
git remote add knowledge-upstream https://github.com/AlexandrShapkin/knowledge.git
git remote -v
```

Не используйте имя `upstream` для базы знаний. Quartz использует `upstream` для репозитория `jackyzha0/quartz`.

Установите зависимости:

```bash
npm ci
```

## 2. Создание рабочей ветки

Получите актуальную основную ветку и создайте отдельную ветку задачи:

```bash
git fetch knowledge-upstream main
git switch -c docs/my-change knowledge-upstream/main
```

Не вносите изменения непосредственно в `main` своего fork.

Рекомендуемые имена:

```text
docs/add-ssh-note
docs/fix-linux-links
fix/markdown-frontmatter
```

Одна ветка и один Pull Request должны соответствовать одной логически связанной задаче.

## 3. Выбор места для заметки

Перед созданием файла:

1. найдите существующие материалы по теме;
2. проверьте, нельзя ли дополнить существующую заметку;
3. выберите ближайший тематический каталог;
4. учтите, что после добавления заметки каталог становится непустым разделом и должен содержать `index.md`;
5. определите, с какими заметками будет связана новая тема.

Новая заметка должна быть атомарной и законченной: раскрывать одну тему и содержать достаточно контекста для самостоятельного чтения.

## 4. Создание заметки

Пользовательские заметки находятся в `content/`.

Минимальный шаблон:

```yaml
---
title: Название заметки
tags:
  - основной-тег
---
```

После frontmatter должно находиться законченное описание одной темы.

Обязательные требования:

- файл сохранён в UTF-8;
- `title` является непустой строкой;
- `tags` является непустым YAML-списком;
- теги написаны в lowercase и не повторяются;
- заметка не дублирует существующий материал;
- заметка имеет хотя бы одну осмысленную входящую или исходящую ссылку.

Дополнительные поля Quartz, например `draft` и `aliases`, разрешены.

## 5. Связывание заметки

Добавьте новую заметку в ближайший тематический `index.md`:

```markdown
- [Диагностика SSH](Диагностика%20SSH.md)
```

При необходимости добавьте в заметку раздел:

```markdown
## Связанные заметки

- [SSH Config](SSH%20Config.md)
- [Проверка DNS](../Сети/Проверка%20DNS.md)
```

Используются только обычные относительные Markdown-ссылки.

Запрещено:

```markdown
[[Диагностика SSH]]
![[Схема сети.png]]
```

Ссылка на заметку должна явно содержать `.md`, использовать URL-кодирование пробелов и вести на существующий файл.

## 6. Изображения и другие вложения

Вложения хранятся в `!assets` той же директории, где находится использующая их заметка.

```text
content/Kubernetes/Ingress/
├── index.md
├── TLS через cert-manager.md
└── !assets/
    └── Схема выпуска TLS-сертификата.png
```

Пример ссылки:

```markdown
![Схема выпуска сертификата](!assets/Схема%20выпуска%20TLS-%D1%81%D0%B5%D1%80%D1%82%D0%B8%D1%84%D0%B8%D0%BA%D0%B0%D1%82%D0%B0.png)
```

Имя файла должно описывать содержимое. Имена вроде `Pasted image`, `image.png`, `screenshot.png`, `file.pdf` и `Untitled` нужно заменить до commit.

Для каталогов, название которых начинается с `!`, `index.md` не требуется. Полностью пустые каталоги также не считаются разделами и не требуют индекса до появления в них заметки или непустого дочернего раздела.

## 7. Локальная проверка

Полная проверка контента:

```bash
npm run content:validate
```

Команда проверяет:

- наличие `index.md` во всех непустых каталогах-разделах;
- пропуск полностью пустых каталогов;
- пропуск служебных каталогов `!…`;
- frontmatter, `title` и `tags`;
- формат и существование внутренних ссылок;
- отсутствие wiki-ссылок;
- связанность заметок;
- размещение и имена вложений.

Запустите локальный просмотр:

```bash
npx quartz build --serve
```

Проверьте итоговую сборку:

```bash
npx quartz build
```

Для изменений механизма синхронизации:

```bash
npm run sync:test
```

Для изменений TypeScript и Quartz:

```bash
npm run check
```

Атомарность, законченность текста, отсутствие смысловых дублей и полезность ссылок нужно проверить вручную.

## 8. Синхронизация ветки

После редактирования выполните:

```bash
npm run sync -- --contributor
```

Перед Git-операциями команда запускает `npm run content:validate`.

Режим контрибьютора:

1. проверяет контент;
2. коммитит локальные изменения;
3. создаёт резервную ветку;
4. получает `knowledge-upstream/main`;
5. объединяет её с текущей веткой;
6. при необходимости объединяет Markdown-конфликты;
7. получает существующую одноимённую ветку из вашего fork;
8. отправляет текущую ветку в `origin` без force push.

Собственное сообщение коммита:

```bash
npm run sync -- --contributor --message "docs(ssh): add host key note"
```

### Пропуск локальной проверки

В аварийной ситуации:

```bash
npm run sync -- --contributor --no-check
```

`--no-check` отключает только предварительную локальную проверку. Проверки Pull Request и deploy остаются обязательными. Не используйте этот флаг как обычный режим работы.

Если команда сообщает, что уже выполняется merge или rebase, сначала завершите или отмените текущую Git-операцию.

## 9. Создание Pull Request

После успешной синхронизации откройте Pull Request:

```text
base repository: AlexandrShapkin/knowledge
base branch: main
head repository: ваш fork
compare branch: ваша рабочая ветка
```

В описании укажите:

- что добавлено или исправлено;
- какие файлы изменены;
- какие проверки выполнены;
- есть ли автоматические решения конфликтов, требующие просмотра.

Название Pull Request должно соответствовать Conventional Commits, например:

```text
docs(ssh): add host key verification note
fix(content): repair broken Kubernetes links
```

## 10. Обновление существующего Pull Request

Если `main` изменилась:

```bash
npm run sync -- --contributor
```

Команда снова получит актуальную `knowledge-upstream/main`, объединит её с вашей веткой и обновит ветку в fork.

## Разрешение конфликтов

Markdown-файлы объединяются трёхсторонним алгоритмом:

- независимые изменения сохраняются;
- `tags`, `aliases` и `cssclasses` объединяются без дублей;
- Markdown-списки объединяются без дублей;
- неоднозначные варианты текста сохраняются оба;
- отчёты записываются в `.git/repository-sync/`.

Для не-Markdown-файлов при конфликте сохраняется локальная версия и создаётся отчёт. Такой результат нужно проверить вручную.

Перед каждой синхронизацией создаётся ветка вида:

```text
backup/repository-sync-<timestamp>
```

## Контрольный список

Перед Pull Request убедитесь, что:

- [ ] каждая новая заметка раскрывает одну тему;
- [ ] текст является законченным и понятным самостоятельно;
- [ ] в каждом Markdown-файле есть `title` и непустой список `tags`;
- [ ] новая заметка связана с графом и добавлена в подходящий `index.md`;
- [ ] в каждом непустом каталоге-разделе есть `index.md`; полностью пустые каталоги не требуют индекса;
- [ ] внутренние ссылки используют Markdown и ведут на существующие файлы;
- [ ] вложения находятся в локальном `!assets`;
- [ ] имена вложений описывают их содержимое;
- [ ] выполнена команда `npm run content:validate`;
- [ ] выполнена команда `npx quartz build`.

## Что не следует делать

Без явной необходимости не используйте:

```bash
npx quartz sync
git push --force
git reset --hard
git clean -fd
```

Не изменяйте пользовательскую автоматизацию внутри `quartz/`, потому что `npx quartz update` может затереть такие изменения.
