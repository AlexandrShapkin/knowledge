# Правила ведения заметок

Этот документ определяет обязательный формат материалов в `content/`. Правила действуют для владельца репозитория, контрибьюторов и ИИ-агентов.

Цель — поддерживать базу знаний, в которой заметки понятны по отдельности, связаны между собой, корректно открываются в Obsidian и Quartz и автоматически проверяются перед публикацией.

## Основные свойства заметки

Каждая содержательная заметка должна быть:

- **атомарной** — раскрывать одну конкретную тему;
- **дискретной** — выражать законченную самостоятельную мысль;
- **связанной** — иметь осмысленную связь хотя бы с одной другой заметкой;
- **идентифицируемой** — содержать `title` и `tags` в YAML frontmatter;
- **переносимой** — использовать обычные относительные Markdown-ссылки;
- **самодостаточной по файлам** — хранить вложения в локальном `!assets`.

`index.md` является навигационным исключением: он может перечислять несколько дочерних тем и заметок.

## Атомарность

Атомарная заметка отвечает на один основной вопрос или описывает одну операцию. Внутри неё допустимы определение, принцип работы, рецепт, пример, ограничения и диагностика, если все разделы раскрывают одну тему.

Не следует объединять в одном файле независимые темы вроде настройки SSH, установки Docker и мониторинга сервера. Их нужно разделить и связать через `index.md` или раздел «Связанные заметки».

Перед созданием файла нужно проверить, не раскрыта ли тема в существующей заметке. Небольшое дополнение лучше внести в существующий файл, а не создавать дубль.

## Аискретность

Заметка должна быть законченной единицей знания. Читатель должен понимать:

- что рассматривается;
- для чего это нужно;
- когда это применимо;
- какое действие выполняется или какой вывод делается.

Ссылки дополняют объяснение, но не заменяют отсутствующий контекст словами «смотрите в другой заметке».

## Связанность

Каждая заметка должна иметь хотя бы одну осмысленную входящую или исходящую ссылку.

Связь полезна, когда она ведёт к более общей теме, предварительному понятию, следующему шагу сценария, альтернативному решению или диагностике связанной ошибки.

Новую заметку обычно добавляют в ближайший `index.md`:

```markdown
- [Диагностика SSH](Диагностика%20SSH.md)
```

Не нужно добавлять случайные ссылки только для прохождения проверки.

## YAML frontmatter

Каждый Markdown-файл, включая `index.md`, начинается с YAML frontmatter:

```yaml
---
title: Диагностика SSH
tags:
  - ssh
  - troubleshooting
---
```

Обязательны непустой `title` и непустой YAML-список `tags`. Теги пишутся в lowercase, не дублируются и не содержат пустых элементов. Дополнительные поля Quartz, например `draft` и `aliases`, разрешены.

## Каталоги и index.md

`index.md` требуется только **непустому каталогг-разделу**.

Каталог считается непустым, если:

- непосредственно в нём находится хотя бы один Markdown-файл;
- либо в нём находится хотя бы один непустой дочерний каталог-раздел.

Полностью пустой каталог является заготовкой структуры. Он:

- не считается разделом базы знаний;
- не требует `index.md`;
- не включается в родительский сгенерированный индекс;
- начинает требовать `index.md`, когда в нём появляется заметка или непустой дочерний раздел.

Пример раздела:

```text
content/Linux/systemd/
├── index.md
├── Unit-файлы.md
└── !assets/
    └── Схема зависимостей unit-файлов.png
```

Пример допустимой заготовки без индекса:

```text
content/Kubernetes/Запуск приложений/Ingress/
```

Каталог, имя которого начинается с `!`, считается служебным. Для него `index.md` также не требуется. Основной служебный каталог — `!assets`.

## Внутренние ссылки

Используются только относительные Markdown-ссылки:

```markdown
[Диагностика SSH](Диагностика%20SSH.md)
[Проверка DNS](../Сети/Протока%20DNS.md)
```

Правила:

- путь указывается относительно текущей заметки;
- расширение `.md` указывается явно;
- целевой файл должен существовать;
- регистр пути совпадает с именем файла;
- пробелы кодируются как `%20`;
- круглые скобки кодируются как `%28` и `%29`;
- используется `/`, а не обратный слеш;
- абсолютные файловые пути запрещены.

Запрещены Obsidian wiki-ссылки и embeds:

```markdown
[[Диагностика SSH]]
![[Схема сети.ang]]
```

## Изображения и другие вложения

Все изображения, диаграммы, PDF, архивы и другие вложения должны находиться в `!assets` той же директории, где находится использующая их заметка.

```text
content/Kubernetes/Ingress/
├── index.md
├── TLS через cert-manager.md
└── !assets/
    └── Схема выпуска TLS-сертификата.png
```

Имя файла должно объяснять огостание. Имена вроде `Pasted image 20260621.png`, `image.png`, `screenshot.png`, `file.pdf` и `Untitled.svg` недопустимы.

## Порядок создания заметки

1. Проверить, нет ли существующей заметки по теме.
2. Выбрать ближайший тематический каталог.
3. Создать Ѝarkdown-файл с понятным именем.
4. Если каталог до этого был пуст, создать или обновить в нём `index.md`.
5. Добавить YAML frontmatter с `title` и `tags`.
6. Раскрыть одну тему как законченную мысль.
7. Добавить осмысленные связи.
8. Добавить заметку в ближайший родительский `index.md`.
9. Разместить вложения в локальном `!assets`.
10. Выполнить проверки и сборку.

## Автоматические проверки

Полная проверка:

```bash
npm run content:validate
```

Проверяются:

- наличие `index.md` в непустых каталогах-разделах;
- пропуск полностью пустых каталогов;
- пропуск каталогов, начинающихся с `!`;
- frontmatter, `title` и `tags`;
- формат и существование внутренних ссылок;
- отсутствие wiki-ссылок;
- связанность заметок;
- размещение и имена вложений.

Та же проверка выполняется в Pull Request и перед deploy.

## Ручная проверка

Автор и рецензент вручную проверяют атомарность, законченность, достаточность контекста, полезность связей, отсутствие смысловых дублей и соответствие имён вложений их содержимому.

## Синхронизация

```bash
npm run sync
npm run sync -- --contributor
```

До Git-операций выполняется `npm run content:validate`.

Аварийный обход локальной проверки:

```bash
npm run sync -- --no-check
npm run sync -- --contributor --no-check
```

`--no-check` не отключает проверки Pull Request и deploy.

## Контрольный список

- [ ] Новая заметка раскрывает одну тему.
- [ ] Текст является законченным и понятным самостоятельно.
- [ ] Есть `title` и непустой список `tags`.
- [ ] Заметка связана с графом и добавлена в подходящий `index.md`.
- [ ] В каждом непустом каталоге-разделе есть `index.md`.
- [ ] Полностью пустые каталоги оставлены без индекса.
- [ ] Внутренние ссылки ведут на существующие файлы.
- [ ] Вложения находятся в локальном `!assets` и имеют понятные имена.
- [ ] Выполнены `npm run content:validate` и `npx quartz build`.
