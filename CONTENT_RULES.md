# Правила ведения заметок

Документ определяет обязательный формат материалов в `content/` для владельца, контрибьюторов и ИИ-агентов.

## Область действия

Публикуемый набор файлов определяется `configuration.ignorePatterns` в `quartz.config.yaml`. Те же шаблоны используют проектные валидаторы.

Из публикации исключены:

- `private`;
- `templates`;
- `.obsidian`;
- `!Meta`.

Файлы из исключённых каталогов не должны использоваться как цели ссылок из публикуемых заметок.

Каталог `!assets` не исключается: его вложения публикуются вместе с заметками.

## Типы страниц Quartz 5

Репозиторий поддерживает:

- `.md` и `.mdx` — текстовые страницы;
- `.canvas` — Obsidian Canvas;
- `.base` — Obsidian Bases.

Требования к YAML frontmatter относятся к Markdown-файлам. Canvas и Bases проверяются сборкой соответствующих Quartz-плагинов.

## Требования к Markdown-заметке

Каждая содержательная заметка должна:

- раскрывать одну конкретную тему;
- быть законченной и понятной самостоятельно;
- иметь осмысленную связь хотя бы с одной другой заметкой;
- содержать `title` и `tags` в YAML frontmatter;
- использовать относительные Markdown-ссылки;
- хранить вложения в локальном `!assets`;
- использовать понятные имена файлов.

`index.md` является навигационным исключением и может объединять несколько дочерних тем.

## YAML frontmatter

Минимальный вариант:

```yaml
---
title: Диагностика SSH
tags:
  - ssh
  - troubleshooting
---
```

Обязательные поля:

- `title` — непустое человекочитаемое название;
- `tags` — непустой YAML-список.

Теги пишутся в lowercase, не дублируются и не содержат пустых элементов.

Поддерживаемые поля Quartz можно использовать по назначению:

- `draft: true` — страница не публикуется;
- `unlisted: true` — страница публикуется по прямому URL, но скрывается из списков, поиска, графа, RSS и sitemap;
- `aliases` — дополнительные имена страницы;
- `description` — описание для метаданных и представления страницы.

Для полностью непубликуемых каталогов используется `ignorePatterns`, а не `draft` или `unlisted`.

## Каталоги и index.md

Quartz 5 умеет генерировать FolderPage без `index.md`. В этом репозитории `index.md` дополнительно используется как явная навигационная страница и как источник связей для графа.

Каждый непустой тематический каталог должен содержать `index.md`. Полностью пустой каталог не требует индекса.

Генерируемые индексы создаются и обновляются командой:

```bash
npm run content:indexes
```

Скрипт не изменяет ручной `index.md`, если в нём нет `generated: true`.

Проверка актуальности:

```bash
npm run content:indexes:check
```

Служебные каталоги с именем, начинающимся с `!`, не считаются тематическими разделами. Основные служебные каталоги:

- `!assets` — публикуемые вложения конкретного раздела;
- `!Meta` — локальные данные Obsidian, полностью исключённые через `ignorePatterns`.

## Внутренние ссылки

Используются относительные Markdown-ссылки:

```markdown
[Диагностика SSH](Диагностика%20SSH.md)
[Проверка DNS](../Сети/Проверка%20DNS.md)
[Проверка конфигурации](Диагностика%20SSH.md#проверка-конфигурации)
```

Правила:

- путь задаётся относительно текущего файла;
- расширение страницы указывается явно: `.md`, `.mdx`, `.canvas` или `.base`;
- цель должна существовать и входить в публикуемый набор Quartz;
- регистр пути должен совпадать с именем файла;
- пробелы и круглые скобки кодируются в URL;
- используется `/`, а не обратный слеш;
- абсолютные файловые пути запрещены.

Quartz 5 поддерживает wiki-links, но проект сохраняет обычные Markdown-ссылки как переносимый и проверяемый контракт. Новые wiki-links и embeds не добавляются.

## Вложения

Изображения, PDF, архивы и другие вложения размещаются в `!assets` той же директории, где находится использующая их заметка.

```text
content/Kubernetes/Ingress/
├── index.md
├── TLS через cert-manager.md
└── !assets/
    ├── Схема выпуска сертификата.png
    └── Пример сертификата.pdf
```

```markdown
![Схема выпуска сертификата](!assets/Схема%20выпуска%20сертификата.png)
[Пример сертификата](!assets/Пример%20сертификата.pdf)
```

Имя вложения должно объяснять его содержимое. Имена вроде `Pasted image`, `image.png`, `screenshot.png`, `file.pdf` и `Untitled` недопустимы.

## Связанность

Заметка должна иметь хотя бы одну полезную входящую или исходящую ссылку. Обычно новая тема добавляется в ближайший `index.md`.

Не следует создавать случайные ссылки только ради прохождения проверки.

## Автоматические проверки

Полная проверка:

```bash
npm run content:validate
```

Она выполняет:

```bash
npm run content:indexes:check
npm run content:links
```

Проверяются:

- соответствие файлов `ignorePatterns` Quartz;
- наличие и актуальность проектных `index.md`;
- YAML frontmatter, `title` и `tags`;
- lowercase и отсутствие дублей в тегах;
- формат, существование и публикуемость целей ссылок;
- отсутствие новых wiki-links;
- связанность Markdown-заметок;
- расположение и имена вложений;
- корректная обработка `.canvas` и `.base` как страниц, а не вложений.

Та же проверка выполняется в Pull Request и перед deployment.

## Ручная проверка

Автор или рецензент вручную оценивает:

- атомарность темы;
- законченность объяснения;
- достаточность контекста;
- отсутствие смысловых дублей;
- полезность связей;
- соответствие имени вложения содержимому.

## Порядок создания заметки

1. Найти существующие материалы по теме.
2. Выбрать ближайший тематический каталог.
3. Создать файл с понятным именем.
4. Добавить frontmatter с `title` и `tags`.
5. Раскрыть одну законченную тему.
6. Добавить полезные связи.
7. Обновить ближайший `index.md` или запустить `npm run content:indexes`.
8. Разместить вложения в локальном `!assets`.
9. Выполнить `npm run content:validate`.
10. Проверить `npx quartz build`.

## Контрольный список

- [ ] Файл входит в публикуемый набор Quartz либо намеренно исключён через `ignorePatterns`.
- [ ] Markdown содержит `title` и непустые `tags`.
- [ ] Заметка раскрывает одну законченную тему.
- [ ] Ссылки относительные и ведут на публикуемые страницы.
- [ ] Заметка связана с графом.
- [ ] Структурные изменения отражены в `index.md`.
- [ ] Вложения находятся в локальном `!assets`.
- [ ] Выполнен `npm run content:validate`.
