# Правила ведения заметок

Этот документ определяет обязательный формат материалов в `content/`. Правила действуют для владельца репозитория, контрибьюторов и ИИ-агентов.

Цель — поддерживать базу знаний, в которой заметки понятны по отдельности, связаны между собой, корректно открываются в разных Markdown-редакторах и автоматически проверяются перед публикацией.

## Требования к каждой заметке

Каждая содержательная заметка должна быть:

- атомарной — раскрывать одну конкретную тему;
- дискретной — выражать законченную самостоятельную мысль;
- связанной — иметь осмысленную связь хотя бы с одной другой заметкой;
- идентифицируемой — содержать `title` и `tags` в YAML frontmatter;
- переносимой — использовать обычные относительные Markdown-ссылки;
- самодостаточной по файлам — хранить вложения в локальном `!assets`.

`index.md` является навигационным исключением: он может перечислять несколько дочерних тем и заметок.

## Атомарность

Атомарная заметка отвечает на один основной вопрос или описывает одну операцию.

Подходящие темы:

- «Как проверить срок действия TLS-сертификата»;
- «Назначение директивы Restart в systemd»;
- «Настройка SSH ProxyJump»;
- «Причины состояния CrashLoopBackOff»;
- «Резервное копирование PostgreSQL».

Внутри одной заметки допустимы определение, принцип работы, рецепт, пример, ограничения и диагностика, если все разделы раскрывают одну тему.

Не следует объединять в одном файле независимые темы вроде настройки SSH, установки Docker и мониторинга сервера. Их нужно разделить на отдельные заметки и связать через `index.md` или раздел «Связанные заметки».

Перед созданием файла нужно проверить, не раскрыта ли тема в существующей заметке. Небольшое дополнение лучше внести в существующий файл, а не создавать дубль.

## Дискретность

Дискретная заметка является законченной единицей знания. Читатель должен понимать:

- что рассматривается;
- для чего это нужно;
- когда это применимо;
- какое действие выполняется или какой вывод делается.

Недопустимы обрывки вроде «дальше указываем этот параметр и перезапускаем сервис», если из файла непонятно, какой параметр, какой сервис и зачем это требуется.

Ссылки дополняют объяснение, но не заменяют отсутствующий контекст словами «смотрите в другой заметке».

## Связанность и граф

Каждая заметка должна иметь хотя бы одну осмысленную входящую или исходящую ссылку.

Связь полезна, когда она ведёт:

- к более общей теме;
- к необходимому предварительному понятию;
- к следующему шагу практического сценария;
- к альтернативному решению;
- к диагностике связанной ошибки.

Новую заметку обычно добавляют в ближайший `index.md`:

```markdown
- [Диагностика SSH](Диагностика%20SSH.md)
```

В самой заметке при необходимости создаётся раздел:

```markdown
## Связанные заметки

- [SSH Config](SSH%20Config.md)
- [Проверка DNS](../Сети/Проверка%20DNS.md)
```

Не нужно добавлять случайные ссылки только для прохождения проверки. Они должны быть полезны читателю.

## YAML frontmatter

Каждый Markdown-файл, включая `index.md`, начинается с YAML frontmatter.

Минимальный вариант:

```yaml
---
title: Диагностика SSH
tags:
  - ssh
  - troubleshooting
---
```

Обязательные поля:

- `title` — непустое человекочитаемое название;
- `tags` — непустой YAML-список тегов.

Дополнительные поля Quartz, например `draft` и `aliases`, разрешены.

Теги:

- пишутся в lowercase;
- не дублируются;
- не содержат пустых элементов;
- описывают предметную область, технологию или назначение заметки.

Неправильно:

```yaml
tags:
  - SSH
  - ssh
  -
```

Правильно:

```yaml
tags:
  - ssh
  - security
```

## Каталоги и index.md

Каждый обычный каталог внутри `content/` обязан содержать `index.md`, даже если в нём пока мало материалов.

Пример структуры:

```text
content/Linux/systemd/
├── index.md
├── Unit-файлы.md
├── Зависимости unit-файлов.md
└── !assets/
    └── Схема зависимостей unit-файлов.png
```

`index.md` описывает назначение раздела и связывает дочерние каталоги и заметки.

Каталог, имя которого начинается с `!`, считается служебным. Для него `index.md` не требуется. Такие каталоги не должны отображаться в навигации и пропускаются автоматической проверкой индексов.

Основной служебный каталог — `!assets`.

## Внутренние ссылки

Используются только относительные Markdown-ссылки:

```markdown
[Диагностика SSH](Диагностика%20SSH.md)
[Проверка DNS](../Сети/Проверка%20DNS.md)
[Проверка конфигурации](Диагностика%20SSH.md#проверка-конфигурации)
```

Правила:

- путь указывается относительно текущей заметки;
- расширение `.md` указывается явно;
- целевой файл должен существовать;
- регистр пути совпадает с именем файла;
- пробелы кодируются как `%20`;
- круглые скобки в пути кодируются как `%28` и `%29`;
- в пути используется `/`, а не обратный слеш;
- абсолютные файловые пути запрещены.

Запрещены Obsidian wiki-ссылки и embeds:

```markdown
[[Диагностика SSH]]
![[Схема сети.png]]
```

Даже если Obsidian понимает этот синтаксис, он не является переносимым контрактом репозитория.

## Изображения и другие вложения

Все изображения, диаграммы, PDF, архивы и другие вложения должны находиться в `!assets` той же директории, где находится использующая их заметка.

```text
content/Kubernetes/Ingress/
├── index.md
├── TLS через cert-manager.md
└── !assets/
    ├── Схема выпуска TLS-сертификата.png
    └── Пример сертификата.pdf
```

Ссылки:

```markdown
![Схема выпуска сертификата](!assets/Схема%20выпуска%20TLS-%D1%81%D0%B5%D1%80%D1%82%D0%B8%D1%84%D0%B8%D0%BA%D0%B0%D1%82%D0%B0.png)
[Пример сертификата](!assets/Пример%20сертификата.pdf)
```

Вложение не должно лежать рядом с Markdown-файлом вне `!assets`, в общем каталоге всей базы или в `!assets` другого раздела.

Имя файла должно объяснять содержимое.

Хорошие имена:

```text
Схема маршрутизации Traefik.png
Диаграмма состояний systemd unit.svg
Пример отчёта PostgreSQL.pdf
```

Недопустимые имена:

```text
Pasted image 20260621.png
image.png
img1.jpg
screenshot.png
file.pdf
Untitled.svg
```

Если вложение используется несколькими заметками одного каталога, допускается общий `!assets` этого каталога. Для заметок из разных каталогов предпочтительны локальные копии.

## Порядок создания заметки

1. Проверить, нет ли существующей заметки по теме.
2. Выбрать ближайший тематический каталог.
3. Убедиться, что в каталоге есть `index.md`.
4. Создать Markdown-файл с понятным именем.
5. Добавить YAML frontmatter с `title` и `tags`.
6. Раскрыть одну тему как законченную мысль.
7. Добавить осмысленные связи.
8. Добавить заметку в ближайший `index.md`.
9. Разместить вложения в локальном `!assets` и переименовать их.
10. Выполнить проверки и сборку.

## Автоматические проверки

Полная проверка контента:

```bash
npm run content:validate
```

Она выполняет:

```bash
npm run content:indexes:check
npm run content:links
```

Проверяются:

- наличие `index.md` в обычных каталогах;
- пропуск каталогов, начинающихся с `!`;
- наличие frontmatter, `title` и `tags`;
- lowercase и отсутствие дублей в тегах;
- формат и существование внутренних ссылок;
- отсутствие wiki-ссылок;
- связанность заметок;
- размещение вложений в локальном `!assets`;
- отсутствие стандартных бессодержательных имён вложений.

Та же проверка выполняется в Pull Request и перед deploy. Ошибка блокирует публикацию.

## Ручная проверка

Скрипт не может надёжно оценить смысл текста. Автор и рецензент вручную проверяют:

- посвящена ли заметка одной теме;
- является ли мысль законченной;
- достаточно ли контекста для самостоятельного чтения;
- полезны ли добавленные связи;
- не дублирует ли файл существующую заметку;
- соответствует ли имя вложения его содержимому.

## Синхронизация

Обычная синхронизация владельца:

```bash
npm run sync
```

Режим контрибьютора:

```bash
npm run sync -- --contributor
```

До Git-операций команда запускает `npm run content:validate`. При ошибке commit, merge и push не выполняются.

Аварийный обход локальной проверки:

```bash
npm run sync -- --no-check
npm run sync -- --contributor --no-check
```

`--no-check` отключает только локальную предварительную проверку. Проверки Pull Request и deploy остаются обязательными.

## Контрольный список перед Pull Request

- [ ] Новая заметка раскрывает одну тему.
- [ ] Текст является законченным и понятным самостоятельно.
- [ ] В каждом Markdown-файле есть `title` и непустой список `tags`.
- [ ] Новая заметка связана с графом и добавлена в подходящий `index.md`.
- [ ] В каждом обычном каталоге есть `index.md`.
- [ ] Внутренние ссылки используют Markdown и ведут на существующие файлы.
- [ ] Все вложения находятся в локальном `!assets`.
- [ ] Имена вложений описывают их содержимое.
- [ ] Выполнена команда `npm run content:validate`.
- [ ] Выполнена команда `npx quartz build`.
